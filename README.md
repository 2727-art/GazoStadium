# 貼り合いスタジアム

好きな画像を5枚用意し、知らない相手と1～10点で採点し合うオンライン画像対戦ゲームです。1on1、2on2チーム対戦、4人バトルロワイヤルを遊べます。

- Firebaseで匿名ログイン・ランダムマッチング・対戦同期・チャットを行い、画像本体はWebRTC DataChannelで対戦相手へ直接送信
- トップ画面の「戦略型1on1対戦」から、匿名自己紹介を読んで相手対策デッキを組むオフライン試作版を2人で遊べます
- トップ画面の参加状況は、通常型1on1・戦略型1on1・2on2・バトルロワイヤルを別々に表示します
- トップ画面の参加状況はRealtime接続を維持せず、公開REST APIから10秒ごとに更新します
- ランキングは画面を開いた時だけREST APIから取得し、トップ画面では常時監視しません

## 起動方法

ES Modulesを使うため、ローカルサーバーから開きます。ビルドやパッケージのインストールは不要です。

```powershell
python -m http.server 4173 --bind 127.0.0.1
```

その後、`http://127.0.0.1:4173/`を開きます。オンライン対戦の動作確認は、1on1なら2環境、2on2・バトルロワイヤルなら4環境のように、保存領域の異なるブラウザまたはブラウザプロファイルで行ってください。

## 対戦ルール

### 戦略型1on1対戦（オフライン試作）

1台の端末を交互に渡して、デッキ構築の情報戦を検証する2人用モードです。Firebaseへの接続やオンラインマッチングは使用しません。

- 各プレイヤーは名前、好みを示す3つの手掛かり、追撃セリフを先に登録
- 追撃セリフは定型文または自由記述を選択。自由記述は1行・最大40文字で、空白時は定型文へ戻る
- 3つの手掛かりは空欄不可で、1つだけブラフとして秘密登録
- 相手の名前を伏せたまま3つの手掛かりを表示し、対戦を受けるか撤退するかを選択
- 対戦を受けたプレイヤーは、メイン5枚と任意のリザーブ最大5枚を相手の好みに合わせて構築
- 両者のデッキ封印後にプレイヤーネームを公開
- 最大5ラウンド。両者がメイン画像を秘密選択し、同時公開後に相手の画像を1～10点で秘密採点
- 採点とは別に、各プレイヤーは1～3点時の「もう1枚見せろ」を2回、9～10点時の「追撃を許可」を1回まで使用可能
- 追加要求を受けた側はリザーブを1枚提示し、メインと追加画像の高い方をラウンドの画像パワーに採用
- 追撃側は事前登録セリフとリザーブを公開し、追撃画像の採点の半分をメインの画像パワーへ加算
- 追加画像から追加要求・追撃は連鎖せず、両者の画像パワー確定後に高い側が同値のダメージを与える
- 最大HP30。HPが0になるか5ラウンド終了で決着し、同HPなら累計画像パワーで判定
- 試合後に両者のブラフ、各ラウンドの画像パワー、平均評価を表示してデッキ構築を振り返れる

画像は既存モードと同じCanvas処理で縮小・WebP化し、ブラウザメモリ内だけに保持します。途中終了・試合終了・ページ終了時に画像URLを解放します。

### 1on1

- 対戦前に各プレイヤーが画像を5枚選択
- 最大HPは30、最大5ラウンド
- 各ラウンドで未使用画像を1枚ずつ秘密裏に選択
- 両者の通信準備完了後に10秒の選択タイマーを開始し、時間切れ時は選択中または未使用の画像を自動ロック
- 両者の選択完了後に画像を交換・公開
- 相手の画像を1～10点で採点
- 高得点側が勝利し、低得点側は高得点と同じ値のダメージを受ける
- 同点ラウンドは両者ノーダメージ
- 8～9点は `CRITICAL!`、10点は `PERFECT!!`
- 9～10点を受けた画像の持ち主について、事前登録した追撃セリフをラウンド結果へ表示（オンライン1on1では演出のみ）
- HPが0になると敗北
- 5ラウンド終了時は、残りHP、合計獲得点、CRITICAL回数、10点回数の順で判定
- すべて同じ場合は引き分け
- ルーム破棄は勝敗・勝率・連勝数に影響しない

### 2on2

- 待機中の4人が揃うと、ランダムに2人ずつのTEAM A / TEAM Bへ自動分配
- 各プレイヤーが画像を5枚選び、最大5ラウンドすべてで4人全員が1枚ずつ提出
- 各ラウンドは20秒のチーム専用作戦チャット後、10秒の秘密選択を実施
- 選択時間切れ時は、選択中または未使用の画像を自動ロック
- 4枚の公開後、各プレイヤーが相手チームの2枚を別々に採点
- 採点時間は20秒。時間切れ時の未採点画像は5点として自動確定
- 画像得点は相手チーム2人の採点平均、チーム得点はチーム2枚の画像得点平均
- 高得点チームが勝利し、相手の共有HP30へ四捨五入したチーム得点分のダメージ
- 画像得点の平均が8点以上で `CRITICAL!`、相手2人がともに10点なら `PERFECT!!`
- 5ラウンド終了時は、共有HP、累計チーム得点、CRITICAL数、PERFECT数の順で判定
- 作戦中は同じチームの2人だけが読めるチャット、画像公開後は4人共通チャットを使用
- ルーム破棄・切断はノーコンテストとなり、2on2戦績とミッションには影響しない

### 4人バトルロワイヤル

- 待機中の4人が揃うと自動的に個人戦を開始
- 各プレイヤーは画像を5枚選び、生存中は毎ラウンド未使用画像を1枚提出
- 画像選択は10秒。時間切れ時は選択中または未使用の画像を自動ロック
- 画像の持ち主は採点確定まで非表示
- 各参加者は自分以外の生存画像を1～10点で採点し、各画像に集まる3票の中央値を画像得点とする
- 毎ラウンド最下位の1人が脱落し、4人→3人→2人→優勝者の順で決着
- 最下位同点の場合は対象者だけが未使用画像を提出するサドンデスを実施
- サドンデスでも同点、または5枚目の場合は累計得点、CRITICAL数、PERFECT数、ルーム単位の決定的抽選の順で脱落者を決定
- 脱落者はその後も審査員と4人共通チャットの参加者として残る
- 退出・切断した本人だけを失格にし、残りの参加者は対戦を継続
- バトルロワイヤル専用戦績として優勝回数、TOP2回数、対戦数、現在・最高連勝を保存

## オンライン機能

- Firebase Authenticationの匿名アカウント
- トップページに1on1・2on2・バトルロワイヤル別の対戦待機中・対戦中人数をリアルタイム表示
- プレイヤーネーム・レート・戦績と任意のXリンクを表示する上位50人のカジュアルランキング（任意参加）
- 毎日0:00（日本時間）に更新される3つのデイリーミッションと、受取式のポイント報酬
- ポイントで対戦チャット用の追加リアクション10種とプレイヤー称号18種を購入できるポイントショップ
- 購入品からリアクションを最大8個、称号を1個装備し、1on1・2on2・バトルロワイヤルのチャットと対戦準備画面へ反映
- ショップ商品は交流・表示専用で、採点・HP・勝敗には影響しない
- 待機中プレイヤーからのランダム1on1マッチング
- オンライン1on1準備画面で追撃セリフを定型文または1行40文字までの自由記述から登録
- 4人をランダムに2チームへ分ける2on2マッチングと、モード別の2on2戦績・レート
- 4人バトルロワイヤルの自動マッチング、個人脱落、サドンデス、専用戦績
- Firebase Realtime Databaseによるルーム、選択完了、採点、ラウンド進行の同期
- 対戦中の2人だけが読めるチャット
- 2on2のチーム専用作戦チャットと、画像公開後の4人共通チャット
- 勝敗数、引き分け数、現在の連勝、最高連勝の保存
- WebRTC DataChannelによる画像の一時的なP2P転送
- Web Audioによるボタン、決定、選択カウントダウン、クリティカル、パーフェクトの効果音
- ヘッダーの`SE ON / SE OFF`切り替え（設定はブラウザに保存）
- 切断時の待機・対戦予約の自動解放
- 古い対戦提案を次回マッチング開始時に清掃

## 画像とデータの扱い

- 選択画像は1on1では最大1600px、2on2・バトルロワイヤルでは4人転送の通信量を抑えるため最大1280pxに縮小してWebPへ変換します。
- Canvasを経由するため、元画像のEXIFなどの付随情報は引き継ぎません。
- 画像本体と画像IDは、両者がそのラウンドの選択をロックするまで相手へ送りません。
- オンライン画像はWebRTCで対戦相手へ直接送り、Firebase Realtime DatabaseやStorageへ保存しません。2on2・バトルロワイヤルでは各ブラウザが残り3人とP2P接続します。
- 対戦終了、ルーム破棄、ページ終了時に画像URLとBlobへの参照を解放します。
- Firebaseへ保存するのは匿名UID、表示名、追撃セリフ、戦績、チーム分け、ルーム状態、点数、チャットなどの対戦メタデータです。追撃セリフは1行40文字以内に制限し、画面ではHTMLとして実行せず文字列として表示します。
- ミッション進捗、所持ポイント、購入済み商品、装備設定は匿名UIDごとの非公開領域へ保存します。装備中の称号IDだけは、チャット送信時に同じルームの参加者へ共有します。
- トップページの参加状況には人数だけを表示し、表示名・匿名UID・ルーム情報は公開しません。閲覧者は人数に含みません。
- ランキング参加者はプレイヤーネーム・レート・戦績を公開します。本人が個別に許可した場合だけ、Xのユーザー名も公開します。ランキングの公開キーには匿名UIDではなくランダムIDを使用します。

ブラウザに表示した画像は、相手によるスクリーンショット、開発者ツール、画面撮影まで技術的に完全防止することはできません。「ゲーム自身が画像ファイルを保存・配布する機能を持たない」という範囲の保護です。

## Firebase構成

このリポジトリはFirebaseプロジェクト `gazostadium` 用に設定済みです。

- Authentication: 匿名ログインを使用
- Realtime Database: マッチング、ルーム同期、チャット、戦績に使用
- Storage: 使用しない
- Firebase Hosting: 必須ではない（GitHub Pagesで公開可能）
- クライアント設定: `firebase-config.js`
- セキュリティルールの管理元: `database.rules.json`

Realtime Databaseのルールは、各ユーザーが自分の待機情報・採点・戦績だけを書けること、ルーム参加者だけがルームと共通チャットを読めることを制約します。2on2の作戦チャットは同じチームだけが読める独立領域です。バトルロワイヤルでは退出者本人、または切断状態を確認できた参加者だけが失格情報を書けます。Firebaseコンソールでルールを変更する場合は、`database.rules.json`も同じ内容に保ってください。

## GitHub Pagesで公開する

1. このフォルダーをGitHubリポジトリへpushします。
2. GitHubのリポジトリで `Settings` → `Pages` を開きます。
3. `Deploy from a branch` を選び、公開ブランチのルートを指定します。
4. 発行されたHTTPS URLへ2つのブラウザ環境からアクセスし、マッチングを確認します。

静的ファイルだけで公開できます。FirebaseのWebクライアント設定は公開される前提の識別情報であり、アクセス制御はRealtime Databaseルールが担当します。

- `index.html`: HTMLシェル

- `audio.js`: ボタン・クリティカル・パーフェクトの効果音

- `team.js`: Firebase/WebRTC 2on2チーム対戦

- `firebase-config.js`: Firebase Webクライアント設定
- `database.rules.json`: Realtime Databaseセキュリティルール

## 現時点の制約

- WebRTCは公開STUNサーバーを使用し、TURNサーバーは未設定です。厳しいNATや組織ネットワーク同士ではP2P接続できない場合があります。3接続が必要な2on2・バトルロワイヤルは、1on1より接続失敗の影響を受けやすくなります。
- 匿名アカウントとオンライン戦績はブラウザのサイトデータに紐づき、サイトデータを消すと新しいユーザー扱いになります。
- 採点の書き換え先はセキュリティルールで本人の領域に限定していますが、完全な不正対策を行う権威サーバーはありません。
- ランキングはFirebaseだけで動くカジュアル版です。戦績とレートは各ブラウザから送信されるため、改ざんを完全には防止できません。
- ランキングのXリンクは自己申告で、アカウント所有者の本人確認は行いません。Xを公開すると匿名性が下がります。
- ミッションとポイントショップもFirebaseだけで動くカジュアル版です。進捗と購入処理はトランザクションで重複を防ぎますが、権威サーバーによる完全な不正対策ではありません。
- 終了済みルームのメタデータは管理用の定期削除をまだ実装していません。画像本体はルームに保存されません。
