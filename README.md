# 貼り合いスタジアム

好きな画像を5枚用意し、相手と1～10点で採点し合う1on1画像対戦ゲームです。

- Firebaseで匿名ログイン・ランダムマッチング・対戦同期・チャットを行い、画像本体はWebRTC DataChannelで相手へ直接送信

## 起動方法

ES Modulesを使うため、ローカルサーバーから開きます。ビルドやパッケージのインストールは不要です。

```powershell
python -m http.server 4173 --bind 127.0.0.1
```

その後、`http://127.0.0.1:4173/`を開きます。オンライン対戦の動作確認は、通常ブラウザとシークレットウィンドウなど、保存領域の異なる2ブラウザで行ってください。

## 対戦ルール

- 対戦前に各プレイヤーが画像を5枚選択
- 最大HPは30、最大5ラウンド
- 各ラウンドで未使用画像を1枚ずつ秘密裏に選択
- 両者の通信準備完了後に10秒の選択タイマーを開始し、時間切れ時は選択中または未使用の画像を自動ロック
- 両者の選択完了後に画像を交換・公開
- 相手の画像を1～10点で採点
- 高得点側が勝利し、低得点側は高得点と同じ値のダメージを受ける
- 同点ラウンドは両者ノーダメージ
- 8～9点は `CRITICAL!`、10点は `PERFECT!!`
- HPが0になると敗北
- 5ラウンド終了時は、残りHP、合計獲得点、CRITICAL回数、10点回数の順で判定
- すべて同じ場合は引き分け
- ルーム破棄は勝敗・勝率・連勝数に影響しない

## オンライン機能

- Firebase Authenticationの匿名アカウント
- トップページにオンライン参加者・対戦待機中・対戦中の人数をリアルタイム表示
- プレイヤーネーム・レート・戦績と任意のXリンクを表示する上位50人のカジュアルランキング（任意参加）
- 待機中プレイヤーからのランダム1on1マッチング
- Firebase Realtime Databaseによるルーム、選択完了、採点、ラウンド進行の同期
- 対戦中の2人だけが読めるチャット
- 勝敗数、引き分け数、現在の連勝、最高連勝の保存
- WebRTC DataChannelによる画像の一時的なP2P転送
- Web Audioによるボタン、決定、選択カウントダウン、クリティカル、パーフェクトの効果音
- ヘッダーの`SE ON / SE OFF`切り替え（設定はブラウザに保存）
- 切断時の待機・対戦予約の自動解放
- 古い対戦提案を次回マッチング開始時に清掃

## 画像とデータの扱い

- 選択画像は最大1600pxに縮小してWebPへ変換します。
- Canvasを経由するため、元画像のEXIFなどの付随情報は引き継ぎません。
- 画像本体と画像IDは、両者がそのラウンドの選択をロックするまで相手へ送りません。
- オンライン画像はWebRTCで相手へ直接送り、Firebase Realtime DatabaseやStorageへ保存しません。
- 対戦終了、ルーム破棄、ページ終了時に画像URLとBlobへの参照を解放します。
- Firebaseへ保存するのは匿名UID、表示名、戦績、ルーム状態、点数、チャットなどの対戦メタデータです。
- トップページの参加状況には人数だけを表示し、表示名・匿名UID・ルーム情報は公開しません。閲覧者は人数に含みません。
- ランキング参加者はプレイヤーネーム・レート・戦績を公開します。本人が個別に許可した場合だけ、Xのユーザー名も公開します。ランキングの公開キーには匿名UIDではなくランダムIDを使用します。

ブラウザに表示した画像は、相手によるスクリーンショット、開発者ツール、画面撮影まで技術的に完全防止することはできません。「ゲーム自身が画像ファイルを保存・配布する機能を持たない」という範囲の保護です。

## Firebase構成

このリポジトリはFirebaseプロジェクト `gazostadium` 用に設定済みです。

- Authentication: 匿名ログインを使用
- Realtime Database: マッチング、ルーム同期、チャット、戦績に使用
- Storage: 使用しない
- Firebase Hosting: 必須ではない（GitHub Pagesで公開可能）
- クライアント設定: `firebase-config.js`
- セキュリティルールの管理元: `database.rules.json`

Realtime Databaseのルールは、各ユーザーが自分の待機情報・採点・戦績だけを書けること、ルーム参加者だけがルームとチャットを読めることを制約します。Firebaseコンソールでルールを変更する場合は、`database.rules.json`も同じ内容に保ってください。

## GitHub Pagesで公開する

1. このフォルダーをGitHubリポジトリへpushします。
2. GitHubのリポジトリで `Settings` → `Pages` を開きます。
3. `Deploy from a branch` を選び、公開ブランチのルートを指定します。
4. 発行されたHTTPS URLへ2つのブラウザ環境からアクセスし、マッチングを確認します。

静的ファイルだけで公開できます。FirebaseのWebクライアント設定は公開される前提の識別情報であり、アクセス制御はRealtime Databaseルールが担当します。

## 主なファイル

- `index.html`: HTMLシェル
- `styles.css`: 画面スタイル
- `app.js`: トップページと共通の画像処理/UI
- `audio.js`: ボタン・クリティカル・パーフェクトの効果音
- `online.js`: Firebase/WebRTCオンライン対戦
- `firebase-config.js`: Firebase Webクライアント設定
- `database.rules.json`: Realtime Databaseセキュリティルール

## 現時点の制約

- WebRTCは公開STUNサーバーを使用し、TURNサーバーは未設定です。厳しいNATや組織ネットワーク同士ではP2P接続できない場合があります。
- 匿名アカウントとオンライン戦績はブラウザのサイトデータに紐づき、サイトデータを消すと新しいユーザー扱いになります。
- 採点の書き換え先はセキュリティルールで本人の領域に限定していますが、完全な不正対策を行う権威サーバーはありません。
- ランキングはFirebaseだけで動くカジュアル版です。戦績とレートは各ブラウザから送信されるため、改ざんを完全には防止できません。
- ランキングのXリンクは自己申告で、アカウント所有者の本人確認は行いません。Xを公開すると匿名性が下がります。
- 終了済みルームのメタデータは管理用の定期削除をまだ実装していません。画像本体はルームに保存されません。
