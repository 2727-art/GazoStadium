# 貼り合いスタジアム

トップページには「10秒音声メーカー」があり、端末のマイク録音または20MB以下の手持ち音声を、戦略型1on1用の先頭10秒・22.05kHz・モノラル・16bit WAVへブラウザ内だけで変換できます。変換後はその場で試聴・保存でき、Firebaseや運営サーバーには送信しません。

好きな画像を用意し、知らない相手と魅力を競うオンライン画像ゲームです。1～10点で戦う1on1・2on2、順位投票で生き残る4人バトルロワイヤル、実際のポイント価値で魅力を競う「推し値市場 / VALUE MARKET」を遊べます。

- Firebaseで匿名ログイン・対戦相手の検索・対戦同期・チャットを行い、画像本体はWebRTC DataChannelで対戦相手へ直接送信
- トップ画面の「戦略型1on1対戦」から、匿名自己紹介を読んで相手対策デッキを組むオンライン対戦を遊べます
- トップ画面の参加状況は、通常型1on1・戦略型1on1・2on2・バトルロワイヤルを別々に表示します
- トップ画面の参加状況はRealtime Databaseを購読し、待機・対戦状態の変化をリアルタイムで更新します
- オンライン総合ランキングは4モードの結果を合算し、デイリー・ウィークリー・マンスリーの選択中期間だけFirebase SDKで取得します。切替日以降の期間はCloud Functionsが検証済み試合から確定します
- ランキング参加設定は4モード共通で、各モードの準備画面とランキング画面から切り替えられます
- 任意のプロフィール画像を同一ブラウザのIndexedDBへ保存し、対戦中だけWebRTCで参加者へ直接転送します
- ランキングコメントも欄を開いた時だけ取得し、投稿・削除時だけ匿名認証付きFirebase SDKで接続します
- 推し値市場の残高移動・売買状態・独立ランキングはCloud FunctionsとCloud Firestoreのトランザクションで確定し、画像と10秒音声はWebRTCで相手へ直接送信します
- 市場成約時だけ売り手へ5%の手数料を課し、買い手へ画像を含まない非譲渡の「推し値証書」を発行します
- 4つのオンライン対戦を正式完走した後、参加者1人へ5・10・20PTの「差し入れ」を1回だけ送れます
- 匿名データを任意のGoogleアカウントへ連携し、同じUIDのポイント・購入品・実績・戦績を別端末から復元できます
- Google保護済みプレイヤーは、毎月300 / 1,500 / 5,000PTを使うVALUE MARKETパトロンへ昇格できます

## 起動方法

ES Modulesを使うため、ローカルサーバーから開きます。ビルドやパッケージのインストールは不要です。

```powershell
python -m http.server 4173 --bind 127.0.0.1
```

その後、`http://127.0.0.1:4173/`を開きます。オンライン対戦の動作確認は、1on1なら2環境、2on2・バトルロワイヤルなら4環境のように、保存領域の異なるブラウザまたはブラウザプロファイルで行ってください。

## 対戦ルール

### 推し値市場 / VALUE MARKET

ポイントの用途がなくなったプレイヤー向けに、画像の魅力を営業し、点数ではなく実際のポイント価値で競う独立エンドコンテンツです。売買はTRPGとしてのロールプレイであり、画像データ・著作権・所有権そのものは移転しません。

- 売り手は画像1枚、出品タイトル、販売価格10～500PT、営業方法を登録し、買い手は購入上限を登録してロール別にマッチング
- 画像はマッチ成立後にWebRTC DataChannelで買い手へ直接送信し、FirebaseやCloud Storageには保存しない
- 買い手は画像と販売価格を無料で確認し、「営業を受ける」を選んだ時点で着手料5PTをFunctionsへ一時預託する。売り手がチャットまたは10秒音声を送り営業完了を確定すると売り手へ支払われ、売り手が完了前に終了した場合は買い手へ返金する
- 無料プレビュー段階を設けることで、売り手がマッチングと即退出を繰り返して着手料だけを得る行為を防ぐ
- 売り手は240文字以内のチャット、またはゲーム用に変換した最大10秒のWAV音声で営業
- 営業終了後、買い手は販売価格で購入、退室、もう1ターン検討から選択
- 購入成立時だけ販売価格の5%（端数切り上げ・最低1PT）を売り手負担の市場手数料として消却する。買い手の支払額は販売価格のままで、売り手は販売価格から手数料を引いた額を受け取る
- 購入成立と同じFirestoreトランザクションで、買い手へ「推し値証書」を1枚発行する。証書は番号、出品タイトル、売り手表示名、成約価格、手数料、営業ターン、成立日時だけを持つ非譲渡コレクションで、画像・音声・チャット・著作権・所有権は含まない
- 追加検討では、売り手が5・10・20PTの内金を提示した時点でFunctionsへ一時預託し、買い手が受け取った場合だけ買い手へ移して次の営業ターンへ進む。辞退・取引中止時は売り手へ戻す
- 営業は最大3ターン。着手料、内金、購入代金はすべてCloud Functionsだけが移動を確定し、一時預託中は受取候補双方の残高上限枠も予約する
- 売買成立時だけ売り手の売上・成約数と買い手の購入額・購入数を更新し、総合ランキングとは別の市場ランキングへ集計
- 同一の売り手・買い手ペアは日本時間の1日につき最初の成立取引だけをランキングへ加算し、相互売買による順位水増しを抑制
- 市場ランキングへ、本人が明示的に公開したXユーザー名と1行40文字以内の「市場プロフィールの一言」を任意表示。Xは自己申告として表示し、自由入力URLではなく検証済みユーザー名から`x.com`リンクを生成する
- 画像の一時判定や音声通報機能は設けず、既存方針どおりユーザー自治で遊ぶ

市場の待機列、アクティブルーム、取引状態、ポイント残高、移動台帳、ロール別実績、証書メタデータはCloud Firestoreに保存します。証書コレクションは本人認証済みCallable Functionsから最新100件まで取得し、クライアントから直接読み書きできません。市場公開プロフィールは売上または購入実績がある本人だけがCallable Functions経由で更新でき、クライアントから統計ドキュメントへ直接アクセスできません。チャット、WebRTCシグナリング、接続状態だけを参加者限定のRealtime Databaseルームへ保存し、画像・音声本体はP2Pメモリ内だけで扱います。

### 対戦後の差し入れ

通常型1on1、戦略型1on1、2on2、4人バトルロワイヤルの最終結果画面では、その試合の自分以外の参加者1人へ5・10・20PTから選んで差し入れできます。

- ポイントを新規発行せず、送信者のFirestoreウォレットから受取人のウォレットへそのまま移すため、全体のポイント総量は増えない
- 送信者1人につき1試合1回だけ。送信後の取消、受取人変更、追加送信は不可
- 自分自身、試合に参加していないUID、ノーコンテスト、全員の結果が揃っていない試合、検証済み試合台帳がない試合はFunctionsが拒否
- 最終結果画面は検証済み試合台帳を確認するまで送信操作を無効化し、相手の確定が遅れた場合は「対戦記録を再確認」から再照会できる
- 金額はFunctions側の固定表だけを許可し、残高不足または受取人のポイント上限到達時は両者の残高を変更しない
- 勝敗、RATE、ランキング、実績、ミッション進捗には影響しない

### データ保護・別端末復元・VALUE MARKETパトロン

トップ画面の「データ保護・市場パトロン」から、現在の匿名データを任意のGoogleアカウントへ連携できます。連携時は新しいゲームデータを作ったり残高を複製したりせず、現在のFirebase Authentication UIDをそのまま維持します。別端末の「別端末のGoogleデータを復元」は、まず現在のゲストUIDへのGoogle連携を試し、すでに保護済みデータがある場合だけ確認後にそのUIDへ切り替えます。選択したGoogleに既存データがなければ空の新規UIDへ移らず、現在のゲストデータをそのまま保護します。端末側とGoogle側のポイントは合算しません。

- Google保護で復元される対象は、UIDに保存されたポイント、購入品と装備、ミッション・期間報酬、実績、オンライン戦績、市場実績、推し値証書、差し入れ送信台帳、パトロン履歴
- プロフィール画像、選択した対戦画像・音声、戦略型プロフィール、SE設定などブラウザ内だけに保存するデータは別端末へ復元しない
- 高額消費はGoogle保護済みアカウントだけが実行可能。日本時間の月ごとに`SUPPORTER 300PT / PATRON 1,500PT / GRAND PATRON 5,000PT`の3段階を用意し、上位へは差額だけを消費
- パトロンは市場ウォレットと商談相手へ表示する名誉バッジで、採点、マッチング、勝敗、ポイント獲得量には影響しない
- 昇格価格はクライアント入力を信用せずFunctions側の固定表から算出し、Firestoreトランザクションと操作IDで二重消費を防止
- 商談・対戦・待機中は、残高競合を避けるためパトロン昇格とアカウント引き継ぎを拒否

Googleを利用できない場合だけ、補助手段として16文字の使い捨て引き継ぎコードを発行できます。コードは10分で期限切れとなり、新しく発行すると以前のコードは無効になります。復元先は対戦、購入、ポイント獲得などをまだ行っていない新しい匿名アカウントに限定し、データをコピーせず発行元と同じUIDへログインします。Google保護後はコードを発行できません。コードはFirestoreへ平文保存せずSHA-256ハッシュだけを保存し、連続試行制限、App Check強制、トークン再利用防止を常時適用します。コードを知った第三者は期限内ならデータへアクセスできるため、SNSやチャットでは共有しないでください。復元後も発行元端末は自動ログアウトされません。

### 戦略型1on1対戦（オンライン）

Firebaseの専用待機キューで採点傾向が近い相手を優先してマッチングし、匿名自己紹介から相手の好みを推測してデッキを構築する2人用オンラインモードです。

- 各プレイヤーは名前、弱点候補3つ、追撃セリフを先に登録し、本当の弱点1つとブラフ2つを決める
- 「アニメ・イラスト」「実写」「どちらも歓迎」から高く評価しやすい画像表現を自己申告する。画像の内容をゲームが自動判定するものではない
- 同じ採点傾向を最優先し、「どちらも歓迎」を次に優先する。異なる傾向同士は待機20秒後に両者が「条件を広げて探す」を選んだ場合だけ成立する
- 追撃セリフは定型文または自由記述を選択。自由記述は1行・最大40文字で、空白時は定型文へ戻る
- 弱点は乱数付きSHA-256コミットとして先に共有し、回答確定後の公開時に改変されていないことを照合
- マッチ成立後は相手の名前を伏せたまま3つの弱点候補を表示し、対戦を受けるか撤退するかを選択
- 匿名紹介からデッキ封印までは名前・プロフィール画像・チャット装飾を伏せた専用チャットで質問やブラフを交わし、弱点推理の材料にできる
- 対戦を受けたプレイヤーは、メイン5枚と任意のリザーブ最大5枚を相手の好みに合わせて構築
- 各画像には任意で音声を1つ添付可能。ブラウザ内で先頭10秒までを22.05kHz・モノラル・16bit WAVへ変換し、音量を正規化する
- 追撃用に音声内の3秒区間を指定でき、弱点看破後の連続追撃では最大3枚・合計最大9秒を順番に再生
- 両者のデッキ封印後にプレイヤーネームを公開
- 名前公開後も全対戦フェイズでチャットを継続し、参加者欄と発言横にP2P受信済みのプロフィール画像を表示
- 最大5ラウンド。両者がメイン画像を秘密選択し、同時公開後に相手の画像を1～10点で秘密採点
- 採点とは別に、各プレイヤーは1～3点時の「もう1枚見せろ」を2回、9～10点時の「追撃を許可」を1回まで使用可能
- 追加要求を受けた側はリザーブを1枚提示し、メインと追加画像の高い方をラウンドの画像パワーに採用
- 追撃側は事前登録セリフとリザーブを公開し、追撃画像の採点の半分をメインの画像パワーへ加算
- 追加画像から追加要求・追撃は連鎖せず、両者の画像パワー確定後に高い側が同値のダメージを与える
- 各ラウンドの結果後、どちらかが一度だけ早期看破を宣言可能。宣言すると両者の秘密回答フェイズが始まり、未宣言ならROUND 3後に強制開始
- 両者の回答確定後に弱点を公開してコミットを検証。看破失敗者は自分に5ダメージを受け、再回答できない
- 看破成功者は残っているリザーブから最大3枚を選び、`4 → 3 → 2`の固定ダメージで連続追撃。両者成功時は同時解決
- 弱点を看破された側は連続追撃結果で「参りました」を選択でき、HP0のSURRENDER KOとして決着する
- 連続追撃でHPを超過したダメージは`OVERKILL`として演出し、ランキングポイントやRATEへは加算しない
- 連続追撃後にHP0なら決着し、それ以外は最大5ラウンドまで継続。同HPなら連続追撃を含む累計画像パワーで判定
- 試合後に両者の本当の弱点・ブラフ・弱点回答・各ラウンドの画像パワー・平均評価を表示

自己紹介、受諾・撤退、デッキ準備、専用チャット、秘密採点、ラウンド進行は参加者だけが読めるRealtime Databaseルームで同期します。チャットは匿名紹介から利用でき、デッキ封印までは相手名・プロフィール画像を表示しません。画像は既存モードと同じCanvas処理で縮小・WebP化し、画像に添付した音声とともにWebRTC DataChannelで対戦相手へ直接転送します。画像・音声本体はFirebaseへ保存せず、途中終了・試合終了・ページ終了時にObject URLを解放します。

### 1on1

- 対戦前に各プレイヤーが画像を5枚選択
- 通常型では「アニメ・イラストが刺さりやすい」「実写が刺さりやすい」「どちらも歓迎」から、自分が高く評価しやすい画像傾向を自己申告
- 同じ傾向同士を最優先し、「どちらも歓迎」は両方の傾向とマッチング。異なる傾向同士は待機20秒後に両者が「条件を広げて探す」を選んだ場合だけ候補にする
- 画像の内容は自動判定せず、選択した傾向はマッチング条件だけに使用して対戦画面には表示しない
- 最大HPは30、最大5ラウンド。サンプル画像を含む場合は1枚につき最大HPが5減り、5枚すべてサンプルならHP5で開始
- 各ラウンドで未使用画像を1枚ずつ秘密裏に選択
- 両者の通信準備完了後に10秒の選択タイマーを開始し、時間切れ時は選択中または未使用の画像を自動ロック
- 両者の選択完了後に画像を交換・公開
- 通常型は気軽に交流できるフリーモードとして、対戦ルーム成立後から試合終了までチャットを常時利用可能
- 相手の画像を1～10点で採点
- 高得点側が勝利し、低得点側は高得点と同じ値のダメージを受ける
- 同点ラウンドは両者ノーダメージ
- 8～9点は `CRITICAL!`、10点は `PERFECT!!`
- 9～10点を受けた画像の持ち主について、事前登録した追撃セリフをラウンド結果へ表示（オンライン1on1では演出のみ）
- HPが0になると敗北
- 5ラウンド終了時は、残りHP、合計獲得点、CRITICAL回数、10点回数の順で判定
- すべて同じ場合は引き分け
- ルーム破棄は勝敗・勝率・連勝数に影響しない

### 2on2

- 待機中の4人が揃うと、ランダムに2人ずつのTEAM A / TEAM Bへ自動分配
- 各プレイヤーが画像を5枚選び、最大5ラウンドすべてで4人全員が1枚ずつ提出
- 各ラウンドは20秒のチーム専用作戦チャットと `TEAM LINK` 作戦選択後、10秒の秘密選択を実施
- 相方と同じ作戦を選ぶと `TEAM LINK` 候補になり、2枚の画像得点が条件を満たすと発動
- `シンクロ` は2枚とも平均7点以上で勝利時+2ダメージ、`エース支援` は片方9点以上・もう片方5点以上で勝利時+3ダメージ、`カバー` は2枚とも平均6点以上で敗北時-2ダメージ
- 選択時間切れ時は、選択中または未使用の画像を自動ロック
- 4枚の公開後、各プレイヤーが相手チームの2枚を別々に採点
- 採点時間は20秒。時間切れ時の未採点画像は5点として自動確定
- 画像得点は相手チーム2人の採点平均、チーム得点はチーム2枚の画像得点平均
- 高得点チームが勝利し、四捨五入したチーム得点へTEAM LINKの追加攻撃・軽減を反映して相手の共有HP30へダメージ
- 画像得点の平均が8点以上で `CRITICAL!`、相手2人がともに10点なら `PERFECT!!`
- 5ラウンド終了時は、共有HP、累計チーム得点、TEAM LINK発動数、CRITICAL数、PERFECT数の順で判定
- 作戦中は同じチームの2人だけが読めるチャット、画像公開後は4人共通チャットを使用
- ルーム破棄・切断はノーコンテストとなり、2on2戦績とミッションには影響しない

### 4人バトルロワイヤル

- 待機中の4人が揃うと自動的に個人戦を開始
- 各プレイヤーは `OPENING`・`MIDDLE`・`FINAL` のメイン3枚とリザーブ2枚を登録
- 1試合に1回だけ、そのラウンドのメイン画像を未使用リザーブへ交換可能
- 画像選択は10秒、順位投票は15秒。時間切れ時は選択中の画像・未選択順位を自動ロック
- 画像の持ち主は投票確定まで非表示で、選択・公開・投票中は4人チャットを停止
- 4人生存時は他の3枚を1位3ポイント・2位2ポイント・3位1ポイント、3人生存時は他の2枚を1位2ポイント・2位1ポイントで順位投票
- 毎ラウンド最下位の1人が脱落し、4人→3人→2人→優勝者の順で決着
- 最下位同点の場合は対象者だけがリザーブ画像を提出するサドンデスを実施
- FINALは脱落した2人が優勝画像へ投票。同票なら過去ラウンドの支持率・1位票・ラウンド首位回数・ルーム単位の決定的抽選で判定
- 脱落者はFINAL前まで勝敗に影響しない観客賞を選び、投票確定後の4人共通チャットにも参加
- 結果画面は8秒後に自動進行し、一時切断は30秒間の再接続猶予後に本人だけを失格として対戦を継続

## 画像とデータの扱い

- 選択画像は1on1では最大1600px、2on2・バトルロワイヤルでは4人転送の通信量を抑えるため最大1280pxに縮小してWebPへ変換します。
- Canvasを経由するため、元画像のEXIFなどの付随情報は引き継ぎません。
- 画像本体と画像IDは、両者がそのラウンドの選択をロックするまで相手へ送りません。
- オンライン画像はWebRTCで対戦相手へ直接送り、Firebase Realtime DatabaseやStorageへ保存しません。戦略型1on1と推し値市場の音声も同じDataChannelで直接送り、Firebaseには保存しません。推し値市場の出品画像は、ブラウザが申告するMIMEだけに依存せず、WebP・PNG・JPEGの実データ形式を送受信時に確認します。戦略型1on1と推し値市場は通常型1on1とは独立したルームでP2P転送し、2on2・バトルロワイヤルでは各ブラウザが残り3人とP2P接続します。
- プロフィール画像は中央を256px正方形のWebPへ変換して同一ブラウザのIndexedDBだけに保存します。通常型・2on2・バトルロワイヤルは対戦開始後、戦略型は「対戦相手、判明」以降だけ表示し、ランキング・トップページ・最終結果には表示しません。
- Xへの結果シェアは`https://gazostadium.anjugames.workers.dev/`を正式URLとして使用し、相手名・具体的な弱点・対戦画像・音声・チャット・匿名UIDを投稿文へ含めません。投稿内容はXの作成画面で本人が確認・編集し、通信エラーやNO CONTESTには共有ボタンを表示しません。
- 対戦終了、ルーム破棄、ページ終了時に対戦画像と受信したプロフィール画像のURL・Blob参照を解放します。自分のプロフィール画像だけは次回対戦用として端末内に残します。
- Firebaseへ保存するのは匿名UID、任意連携したGoogleの認証識別子、表示名、追撃セリフ、トップメッセージ、通常型・戦略型1on1待機中の自己申告による採点傾向と検索範囲、戦略型1on1の弱点候補・弱点コミット・回答確定後の弱点公開値、2on2のTEAM LINK作戦、モード別戦績、非公開の総合戦績、公開設定時の期間別総合ランキング戦績、ランキングコメント、通常型1on1のサンプル枚数・開始HP、チーム分け、ルーム状態、点数、チャット本文・スタンプIDなどの対戦メタデータです。Googleの名前・メールアドレスはゲーム内表示名、対戦、ランキングへ使用・公開しません。通常型・戦略型の採点傾向は待機キューから退出した時点で削除し、共有ルームへ複製しません。スタンプ画像本体は静的WebPでありFirebaseへ保存・送信しません。トップメッセージの公開キーには匿名UIDではなく専用のランダムIDを使用します。戦略型チャットには送信者UID、本文、匿名探索中か対戦中か、ラウンド、送信時刻を保存し、正体判明後の発言だけ装備中の装飾IDも保存します。プロフィール画像は保存しません。戦略型の待機キューにはUIDと時刻・待機状態・採点傾向・検索範囲だけを置き、弱点番号そのものは両者が回答するまで共有ルームへ保存しません。トップメッセージは1行30文字以内、追撃セリフは1行40文字以内、各チャットとランキングコメントは1行80文字以内に制限し、画面ではHTMLとして実行せず文字列として表示します。
- ミッション進捗、期間戦績報酬、報酬済み試合ID、実績進捗・解除済み実績、サーバーランキング参加状態・期間戦績・確定実績、所持ポイント、差し入れ台帳、推し値証書の正本はCloud Firestoreのサーバー専用領域へ保存します。対戦終了時は参加者本人だけが一度書ける結果申告をルームへ残し、Cloud Functionsが全参加者の申告整合、参加者・プロフィール・承認人数、本人が書いた採点証跡、12時間以内のアクティブルームを確認した場合だけ加算します。差し入れはこの検証済み試合台帳と両者の参加を再確認し、送信者・モード・ルームの一意台帳と両ウォレットを同じトランザクションで更新します。重複防止IDは件数で削除しないFirestore台帳へ永続保存します。既存の検証済み月間戦績から通算・モード別・敗北・対戦日数を遡及しますが、試合順を復元できない連敗は実績導入後から数えます。Realtime Databaseのミッション・期間戦績・残高・サーバーランキングは表示用ミラーで、クライアントから変更できません。購入済み商品と装備設定はRealtime Databaseの匿名UIDごとの非公開領域へ保存します。装備中の称号・チャット背景・チャットフレーム・購入スタンプのIDだけは、チャット送信時に同じルームの参加者へ共有します。戦略型1on1の匿名偵察中は装飾IDを送信せず、無料スタンプIDだけを許可します。
- トップページの参加状況には人数・件数だけを表示し、閲覧者は含みません。推し値市場は参加ごとに発行するランダム公開ID、ロール別の最終通信時刻、競合防止用の状態世代だけを公開領域へ反映し、60秒以内に通信した待機者と、売り手・買い手の双方が60秒以内に通信した商談だけを集計します。購入者が自分で投稿したトップメッセージだけは表示名・称号とともに公開しますが、匿名UID・ルーム情報・プロフィール画像は表示しません。
- ランキング参加者はプレイヤーネーム・総合RATE・動的算出するクラス・選択期間の総合戦績、合計ポイント、モード別ポイントを公開します。基本クラス境界は`Beginner 100–1024 / Great 1025–1049 / Expert 1050–1099 / Veteran 1100–1149 / Ultra 1150–1199 / Master 1200–1299 / Legend 1300–3000`です。`BEYOND`はRATE 1400以上かつ当月マンスリー総合ランキング10位以内の間だけLegendに代わって表示し、Firebaseへ固定保存しません。すべてのクラスは待機室・対戦・チャット・結果画面には表示しません。サーバー化時の総合RATEは旧RATEを初期値として引き継ぎ、以後は検証済み正式対戦だけで更新します。デイリーは毎日0:00、ウィークリーは毎週月曜0:00、マンスリーは毎月1日0:00（日本時間）に新しい期間キーへ切り替わります。本人が個別に許可した場合だけ、Xのユーザー名と自分宛てのコメントも公開します。

ブラウザに表示した画像は、相手によるスクリーンショット、開発者ツール、画面撮影まで技術的に完全防止することはできません。「ゲーム自身が画像ファイルを保存・配布する機能を持たない」という範囲の保護です。

## Firebase構成

このリポジトリはFirebaseプロジェクト `gazostadium` 用に設定済みです。

- Authentication: 匿名ログインを基本とし、任意のGoogle連携と引き継ぎ用Custom Tokenで同一UIDを別端末へ復元
- Realtime Database: 各対戦モードのマッチング、ルーム同期、チャット、戦績、トップページ用の匿名化済み参加状況と、推し値市場のチャット・WebRTCシグナリングに使用
- Cloud Firestore: 権威ウォレット、サーバー検証済みミッション・期間戦績・試合台帳、差し入れ台帳、推し値市場の待機列・取引状態・移動台帳・独立ランキング・証書、月間パトロン、ハッシュ化した短期引き継ぎコードに使用
- Cloud Functions（`us-central1`）: `nam5`のCloud Firestoreに近接し、ポイント報酬・ショップ購入・市場取引・成約手数料・証書発行・対戦後差し入れ・パトロン消費・使い捨て引き継ぎをサーバー権限で確定
- Storage: 使用しない
- Firebase Hosting: `https://gazostadium.web.app` で本番公開
- クライアント設定: `firebase-config.js`
- セキュリティルールの管理元: `database.rules.json` / `firestore.rules`
- Firebase CLI設定: `.firebaserc` / `firebase.json`（既定プロジェクト `gazostadium`）

Realtime Databaseのルールは、各ユーザーが自分の待機情報・採点・順位投票・モード別戦績・総合戦績・移行前の期間別ランキング・一度限りの対戦結果申告だけを書けること、各ルームの参加者だけがルーム情報を読めることを制約します。期間キーがサーバー切替日以降のランキング、ポイント残高、デイリー進捗、購入済み商品、期間戦績報酬、ランキング実績、報酬済みルーム台帳へのクライアント直接書き込みは禁止し、Cloud FunctionsのAdmin SDKだけが更新します。推し値市場の公開参加状況もクライアント書き込みを禁止し、Functionsが参加単位のランダムIDと集計に必要なロール・最終通信時刻・状態世代だけを更新します。推し値市場のRealtime Databaseルームは、取引中の参加者本人のチャット・接続状態と、相手宛てWebRTCシグナルだけを書けます。終了後は新規投稿を拒否し、シグナル削除とオフライン化だけを許可します。市場の取引状態・ウォレット・ランキングはFirestore Rulesでクライアント書き込みを全面禁止し、任意公開プロフィールもFunctionsが本人の実績ドキュメントへだけマージします。ランキング応答には匿名UIDを含めず、表示名・実績・検証済みXユーザー名・1行の一言だけを返します。Firebaseコンソールでルールを変更する場合は、リポジトリ内の両Rulesも同じ内容に保ってください。

## Firebaseへデプロイする

初回だけFirebaseコンソールでCloud Firestoreの本番データベースを作成し、`functions`フォルダーで依存関係をインストールします。

```powershell
npm --prefix functions install
npx --yes firebase-tools deploy --only firestore,database --non-interactive
npx --yes firebase-tools deploy --only functions --non-interactive
npx --yes firebase-tools deploy --only hosting --non-interactive
```

初回はこの順番を守り、先にクライアントからの残高・デイリー進捗・期間戦績報酬への直接書き込みをRulesで閉じてから、Functionsと新しいフロントエンドを公開します。旧版のRealtime Databaseにある残高と購入済み商品は最初のFunctions利用時に一度だけFirestoreへ移行しますが、旧ルール下で作られた値の正当性を後から完全には判別できません。公開前にRealtime Databaseをエクスポートして異常な高残高・大量商品を確認してください。デイリー進捗と期間戦績は安全のため旧値を移行せず、新しいサーバー台帳から開始します。

サーバーランキング版を既存本番へ追加するときは、先に`economyAction`を公開し、その直後にRulesとフロントエンドをまとめて公開します。新しいフロントエンドは登録用Callableを必要とし、新しいRulesは切替日以降のクライアントランキング書き込みを拒否するため、この順番を変えないでください。

```powershell
npx --yes firebase-tools deploy --only functions:economyAction --non-interactive
npx --yes firebase-tools deploy --only database,firestore:rules,hosting --non-interactive
```

最初の切替日までに公開しない場合は、過去日付のまま公開しないでください。`functions/server-ranking.js`、`online.js`、`database.rules.json`、テスト、上記READMEの切替キーを、公開後に到来する次の日次・月曜・月初へ同時に繰り下げます。サーバー版公開前の試合はランキングへ遡及加算しません。

Google保護を本番で利用する前に、Firebase ConsoleのAuthenticationで匿名認証を維持したままGoogleプロバイダを有効化し、サポートメールを設定します。WorkerやGitHub Pagesで公開する場合は、それぞれのHTTPSホスト名をAuthenticationの承認済みドメインへ追加してください。`accountTransfer`は`consumeAppCheckToken: true`を常時指定しているため、第2世代Functionsの実行サービスアカウントへFirebase App Check Token Verifierロールが必要です。Custom Token作成に既定外のサービスアカウントを使う構成では、Firebase Admin SDKのトークン署名に必要なIAM権限も確認してください。

App CheckはreCAPTCHA Enterpriseを使用します。`firebase-client.js`が全モード共通のFirebase AppとApp Checkを初期化し、その後`firebase-services.js`がAuth・Realtime Database・Firestore・Functionsを生成します。サイトキーは`firebase-config.js`の`appCheckRecaptchaEnterpriseSiteKey`へ設定します。初回リリースでは強制せず、3本番ドメインで最新版を配信してから、App Checkの`Verified`比率とFunctionsログを監視します。localhostから本番バックエンドへ接続してデバッグトークンを使う場合だけ`?appCheckDebug=1`を付けます。`?firebaseEmulators=1`と`?marketPreview=1`では本番App Checkを初期化しません。

強制適用は「経過日数」ではなく、実通信の確認結果を基準にします。各ドメインで通常型・戦略型・2on2・ロイヤル・推し値市場、ショップ、デイリー、期間報酬、ランキングコメントを能動テストし、正規通信が概ね99%以上`Verified`で、`Outdated client`と`Invalid`に正規ユーザーが含まれないことを確認します。サービスごとの推奨順は次のとおりです。

1. Firestoreを強制し、反映後15分以上待ってVALUE MARKETを再確認する
2. `functions/app-check-rollout.js`の段階を`rankings_enforced`へ進め、`firebase deploy --only functions:valueMarketRankings`で個別デプロイする
3. 段階を`market_migration`へ進め、`valueMarketQueue`と`valueMarketAction`を再デプロイする。未検証クライアントの新規参加・heartbeatを止め、検証済みの新規ルームを保護しながら、旧ルームの完了経路は残る
4. `valueMarketRooms`の非終端ルームを一覧し、`appCheckVerified: true`でないもの（フィールド未設定を含む）が0件になるまで待つ
5. 段階を`market_enforced`へ進めて`valueMarketAction`と`valueMarketQueue`を強制し、その後`economy_enforced`へ進めて`economyAction`を最後に強制する
6. 全オンラインモードが依存するRealtime Databaseを最後に強制する

Callableは`APP_CHECK_ROLLOUT_STAGE`の1値から関数別の強制設定を導出し、市場関数の片側だけを強制するような不整合な設定を作らない構成です。市場待機列と市場ルームには`appCheckVerified`を記録するため、移行前から作られた取引を判別できます。段階は必ず記載順に進め、各段階を低利用時間帯に行い、反映後15分以上のスモークテストと24時間程度の監視を挟みます。問題があればFirestore・Realtime Databaseはコンソールで監視へ戻し、Callableは段階を直前の値へ戻して対象関数だけ再デプロイします。

本番URL: `https://gazostadium.web.app`

ローカルでAuthentication・Functions・Firestore・Realtime Databaseをまとめて検証する場合は、別ターミナルで次を起動し、`http://127.0.0.1:4173/?firebaseEmulators=1`を開きます。このクエリはlocalhostでだけ有効です。誤って本番データへ接続しないよう、必ず`demo-`で始まる架空プロジェクトIDを使用します。Database・Firestore Emulatorの起動にはJavaが必要です。

```powershell
npx --yes firebase-tools emulators:start --only auth,functions,firestore,database --project demo-gazostadium
```

Javaを用意できない環境で画面レイアウトだけを確認する場合は、`?marketPreview=1`で500PTのローカルプレビューを開けます。トップからデータ保護画面を開くとゲスト状態を確認でき、`?marketPreview=1&accountPreview=protected`ではGoogle保護済み・6,400PTの表示を確認できます。プレビューはFunctionsやデータベースへ接続せず、認証・取引・ポイント消費・引き継ぎ確定テストには使用しません。

## GitHub Pagesで公開する

1. このフォルダーをGitHubリポジトリへpushします。
2. GitHubのリポジトリで `Settings` → `Pages` を開きます。
3. `Deploy from a branch` を選び、公開ブランチのルートを指定します。
4. 発行されたHTTPS URLへ2つのブラウザ環境からアクセスし、マッチングを確認します。

GitHub Pagesにはフロントエンドの静的ファイルだけを配置できますが、オンライン対戦と推し値市場を動かすには、同じFirebaseプロジェクトへRealtime Database Rules・Firestore Rules/Indexes・Cloud Functionsを別途デプロイする必要があります。FirebaseのWebクライアント設定は公開される前提の識別情報であり、アクセス制御は両データベースのRulesとCloud Functionsが担当します。

## 現時点の制約

- WebRTCは公開STUNサーバーを使用し、TURNサーバーは未設定です。厳しいNATや組織ネットワーク同士ではP2P接続できない場合があります。3接続が必要な2on2・バトルロワイヤルは、1on1より接続失敗の影響を受けやすくなります。
- Google保護または引き継ぎを行っていない匿名アカウントはブラウザのサイトデータに紐づき、サイトデータを消すと復元できません。Google保護後はUIDに保存したオンラインデータを復元できますが、端末保存したプロフィール画像・対戦素材・音声・設定は対象外で再登録が必要です。
- 採点の書き換え先はセキュリティルールで本人の領域に限定していますが、通常対戦の採点・勝敗自体は完全な権威サーバー方式ではありません。
- サーバー切替日より前に始まったオンライン総合ランキングは従来のカジュアル表示で、期間終了まで現在順位を維持します。この移行前データへランキング実績やPTを後付けしません。切替日以降はCloud Functionsが検証済み試合台帳から期間戦績とRATE変動を確定し、Realtime Databaseには匿名UIDを含まない公開ミラーだけを書きます。ランキング実績は名誉記録でPTを発生させません。ポイントを発生させるデイリー・期間戦績と、推し値市場のポイント移動・独立ランキングもCloud Functionsで確定します。
- サンプル画像のHPハンデもクライアントとRealtime Database Rulesで整合性を確認するカジュアル版です。改造したクライアントがサンプル画像を通常画像として偽装することまでは完全に防止できません。
- ランキングのXリンクは自己申告で、アカウント所有者の本人確認は行いません。Xを公開すると匿名性が下がります。
- ランキングコメントは公開情報です。URLと改行は画面側で拒否し、ルールでも1行80文字以内に制限しますが、自動モデレーションや通報機能はまだありません。
- トップメッセージも公開情報です。URLと改行は画面側で拒否し、ルールでも1行30文字以内に制限します。閲覧者単位の非表示には対応しますが、自動モデレーションや運営向け通報画面はまだありません。
- 採点系ミッションと期間戦績は、Functionsが全参加者の一度限りの結果申告、本人が書いた採点証跡、実在ルームの参加・完走・重複を検証します。ただし匿名認証である以上、複数端末・複数匿名アカウントを本人確認なしに完全排除することはできません。報酬受取・ショップ購入・市場内の残高移動はFirestoreウォレットの上限・価格・取引状態・冪等操作IDを検証します。クライアントから残高、デイリー進捗、購入済み商品、期間戦績報酬を直接書き換えることはできません。
- 推し値市場の同一ペア日次除外は単純な相互売買のランキング水増しを抑えますが、複数の匿名アカウントを用いた組織的な取引までは完全に識別できません。App Check有効化とFunctionsログの監視を推奨します。
- 終了済みルームのメタデータは管理用の定期削除をまだ実装していません。画像本体はルームに保存されません。
